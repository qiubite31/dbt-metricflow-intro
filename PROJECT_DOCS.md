# dbt MetricFlow Project Overview

This document provides a comprehensive guide to the dbt-metricflow-intro project, which uses dbt (data build tool) 1.9+, MetricFlow (dbt's Semantic Layer), and **DuckDB** to model customer and order data, enabling advanced, code-driven analytics.

## Project Structure

The project follows a modern dbt directory structure with emphasis on the semantic layer:

`	ext
dbt-metricflow-intro/
├── dbt_project.yml           # Main project configuration file
├── profiles.yml              # Database connection profile configuration (DuckDB)
├── data.duckdb               # DuckDB database file (created by setup_data.py)
├── setup_data.py             # Python script to generate sample data and create the database
├── run_dbt_tasks.py          # Python script to automate dbt run, test, and docs generate
├── query_data.py             # Python script to verify and query data in DuckDB
├── semantic_query.py         # Python script to programmatically query MetricFlow SQL and fetch results
├── models/                   # Contains SQL model files (physical layer)
│   ├── schema.yml            # Schema definitions and tests for models
│   ├── stg_customers.sql     # Staging model for customers
│   ├── stg_orders.sql        # Staging model for orders
│   ├── metricflow_time_spine.sql # Essential time spine model for MetricFlow time queries
│   ├── metricflow_time_spine.yml # Time spine definition
│   ├── customer_order_detail.sql # Detailed order view (materialized as table)
│   └── customers_with_orders.sql # Final model joining customers and orders
├── semantic_models/          # Contains MetricFlow Semantic Models (.yml)
│   ├── customers.yml         # Defines dimensions and entities for the customer model
│   └── orders.yml            # Defines dimensions, entities, and measures for the order model
├── metrics/                  # Contains MetricFlow Metrics (.yml)
│   └── order_metrics.yml     # Defines metrics like total_orders, total_revenue, etc.
├── logs/                     # dbt log files
└── target/                   # Compiled SQL files and run artifacts (generated by dbt)
`

## Key Files Description

### Configuration

*   **dbt_project.yml**:
    *   **Project Name**: my_duckdb_project
    *   **Profile**: my_duckdb_profile
    *   **Model Paths**: models, semantic_models, metrics
    *   **Default Materialization**: iew

*   **profiles.yml**:
    *   Defines the my_duckdb_profile configuration.
    *   Connects to the data.duckdb database located in the project root.
    *   Target schema: main

*   **setup_data.py**:
    *   A Python script that initializes the DuckDB backend data.duckdb.
    *   Creates two raw tables: aw_customers and aw_orders with sample data.

### Physical Layer Models (models/)

The models directory contains the transformation logic written in SQL.

1.  **stg_customers.sql** & **stg_orders.sql**:
    *   **Purpose**: Cleans and renames raw data. Casts order_date as DATE for DuckDB compatibility.
2.  **customers_with_orders.sql** & **customer_order_detail.sql**:
    *   **Purpose**: Creates joined tables with aggregated statistics and detailed granular views.
3.  **metricflow_time_spine.sql**:
    *   **Purpose**: Generates consecutive date rows using DuckDB's ange() function. Crucial for MetricFlow to construct valid time-series queries.

### Semantic Layer (semantic_models/ & metrics/)

This directory powers the dbt Semantic Layer (MetricFlow).

1.  **semantic_models/orders.yml**:
    *   Identifies the underlying model (ef('stg_orders')).
    *   Defines **Entities** (like order_id, customer_id as foreign keys).
    *   Defines **Dimensions** (like order_date, status).
    *   Defines **Measures** (like evenue, orders).
2.  **semantic_models/customers.yml**:
    *   Maps to stg_customers. Defines customer_id as a primary entity.
3.  **metrics/order_metrics.yml**:
    *   Defines the actual business metrics you can query using mf query.
    *   Examples: 	otal_orders, 	otal_revenue, verage_order_value.

### Execution Scripts

*   **query_data.py**:
    *   Verifies physical layer transformations directly against DuckDB using Pandas.
*   **semantic_query.py**:
    *   Uses Python's subprocess to trigger MetricFlow CLI (mf query, mf list).
    *   Extracts compiled SQL dynamically and executes it against DuckDB to return multi-dimensional aggregated results. 

## Getting Started

1.  **Initialize Data**: Run python setup_data.py to create a fresh data.duckdb.
2.  **Run dbt Automation**: Execute python run_dbt_tasks.py or manually run dbt build to compile the models and the time spine.
3.  **Verify Physical Data**: Execute python query_data.py to check standard DuckDB tables.
4.  **Query Semantic Layer**: Execute python semantic_query.py to programmatically fetch metric data, or use the MetricFlow CLI (mf list metrics / mf query --metrics total_orders --group-by metric_time).
